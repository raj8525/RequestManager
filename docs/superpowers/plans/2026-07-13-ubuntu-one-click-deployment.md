# Ubuntu One-Click Deployment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver one Bash entry point that installs or updates RequestManager on Ubuntu through GitHub and Docker, and safely synchronizes the complete local SQLite-plus-screenshots backup to that server over SSH.

**Architecture:** A revision-tagged Docker image owns Node.js 24, Next.js and every existing operational command, while one bind-mounted host directory owns all runtime data. `scripts/deploy-ubuntu.sh` provides public `deploy`, `sync`, `status` and `logs` commands plus a guarded remote restore command; it delegates consistency to the existing application backup/restore implementation rather than copying SQLite directly.

**Tech Stack:** Bash 4+, Docker Engine, Git, OpenSSH, Node.js 24 container, Next.js 16, SQLite, Vitest, existing RequestManager backup manifest v2.

## Global Constraints

- The server is one currently Docker-supported Ubuntu LTS host on `amd64` or `arm64`; reject non-Ubuntu server deployment.
- Run exactly one RequestManager application container and one SQLite writer.
- Default repository is `https://github.com/raj8525/RequestManager.git`, default revision is `main`, and default published port is `13001`.
- Persistent host paths are `/opt/request-manager`, `/var/lib/request-manager`, and `/etc/request-manager/request-manager.env`.
- Never put GitHub tokens, SSH secrets, administrator passwords, `.env*`, `data/`, backups or browser artifacts in Git or the Docker image.
- Never use `eval`, disable SSH host-key checking, or directly copy a live SQLite database.
- Every restore must use the existing manifest/hash/integrity/foreign-key/migration-journal verification and atomic swap.
- Every task follows red-green-refactor and ends with exact staging, an intentional commit, and `git push origin main`.

- [ ] **Step 3: Implement strict parsing and dependency/bootstrap functions**

Start with:

```bash
#!/usr/bin/env bash
set -Eeuo pipefail
IFS=$'\n\t'
umask 027

readonly DEFAULT_REPO="https://github.com/raj8525/RequestManager.git"
readonly DEFAULT_REVISION="main"
```

Implement a `case` parser with no `eval`. Validate origins with `http://` or `https://` plus a non-empty host, Git refs with `^[A-Za-z0-9._/-]+$` excluding `..`, ports as decimal `1..65535`, SSH targets as `[user@]host` without whitespace/shell metacharacters, and repository URLs as credential-free HTTPS or SSH Git URLs. `ensure_server_dependencies` reads `/etc/os-release` (or the test override), requires `ID=ubuntu`, installs Git/curl/CA certificates when absent, and installs Docker from Docker's official Ubuntu apt repository only when `docker` is absent. It must never curl a script into an interactive shell.

`ensure_checkout` clones the configured repository into the install root or validates the existing `origin`, clean worktree and `.git`; then fetches and checks out the requested ref. `build_revision_image` resolves the full commit, tags `request-manager:<12-char-sha>`, and completes `docker build` before any container stop.

`write_runtime_environment` atomically writes mode `0600` values for `/app/data` paths, `APP_ORIGIN`, `SECURE_COOKIES` derived from HTTPS, and `TRUST_PROXY_HEADERS=false`. Never write administrator credentials.

- [ ] **Step 3: Implement local sync and guarded remote receive**

`command_sync` runs without root, locates the repository from the script path, requires a clean tracked worktree, resolves `HEAD`, fetches origin and proves the commit is reachable from an origin ref. It creates a `mktemp -d` root, sets only `BACKUP_PATH` to `<temp>/backups`, invokes `npm run ops:backup`, and requires exactly one newly created backup directory. A trap removes only that validated temporary root.

Before SSH activity, `confirm_sync` prints source database path, attachment path, destination, revision and backup directory. Accept only typed `yes` or `REQUEST_MANAGER_SYNC_CONFIRM=yes`.

Remote deployment is invoked as argument-safe SSH command using the installed script and `--revision <sha> --skip-admin`. Upload uses `scp -P <port> -r -- <backup-dir> <target>:<random-private-path>` and retains normal SSH host verification.

`command_receive_backup` requires root, accepts only a generated basename matching `^request-manager-[A-Za-z0-9._-]+$`, rejects symlinks and non-directories, moves it to a fresh incoming path, creates a server-side protection backup, stops the container, runs current-image `ops:restore` with both flags, starts the container, runs `ops:attachments:check`, and health-checks. If restore itself fails, restart unchanged current data. If integrity or health fails after restore, stop and restore the protection backup before restarting. Remove incoming data only after all gates pass.
