# 需求沟通截图与完成说明设计

**状态：** 已批准  
**日期：** 2026-07-22

## 1. 目标

为客户需求补齐开发者面向客户沟通时的截图能力，并增加可持续补写的完成说明。统一调整需求进度标签颜色，发布时严格保护公网服务器上的现有用户数据。

本次包含：

- 开发者公开备注支持选择、拖放和 `Ctrl+V` 粘贴截图。
- 开发者在“确认与澄清”中提问时支持选择、拖放和 `Ctrl+V` 粘贴截图。
- 客户回复澄清时使用同一套截图操作和展示能力。
- 开发者把需求设为完成时可填写可选完成说明并附带截图。
- 完成说明在完成后仍可补写或修改；需求重新变为未排期或已排期后仍保留并显示。
- 进度标签改为未排期灰色、已排期蓝色、完成绿色。
- 每次公网代码更新前必须成功创建远端完整备份，禁止用本机数据覆盖公网数据。

## 2. 明确不包含

- 不增加普通文件、视频或非图片附件。
- 不允许客户编辑完成说明。
- 不把公开备注和澄清合并为一种沟通模型。
- 不实现完成说明版本列表；每条需求保存一份当前完成说明，公开操作历史只记录变更事件，不复制正文。
- 不改变现有进度枚举、暂停、归档和恢复规则。

## 3. 数据模型

采用独立业务表并复用现有受保护截图存储，不使用多态附件关联。

### 3.1 沟通截图

新增：

- `public_remark_attachments`：关联公开备注、需求、上传人和文件元数据。
- `clarification_message_attachments`：关联澄清消息、需求、上传人和文件元数据。

每张截图保存随机存储名、原始文件名、MIME、字节数、SHA-256 和创建时间。外键明确指向所属消息与需求，存储名全局唯一。查询时通过需求重新验证实时账号和项目权限。

### 3.2 完成说明

新增：

- `completion_notes`：每条需求最多一行，包含正文、最后编辑开发者和创建/更新时间。
- `completion_note_attachments`：关联完成说明、需求、上传人和文件元数据。

正文允许为空，截图也允许为空。没有正文和截图时可以不创建说明行，但需求仍能正常进入完成状态。已有完成说明在需求离开完成状态后继续保留。

## 4. 领域规则

### 4.1 公开备注与澄清

- 公开备注仍只允许开发者在正常需求中添加，正文必填。
- 开发者提问和客户回复仍遵守现有待回复状态机，正文必填。
- 每条新消息最多 8 张图片，单张不超过 10 MiB，合计不超过 30 MiB，只接受 PNG、JPEG 和 WebP。
- 消息、附件元数据和需求版本更新在同一个 SQLite `IMMEDIATE` 事务中完成。
- 文件先暂存并校验魔数，事务内提交到随机路径；任何失败都回滚数据库并清理本次文件。
- 幂等重放必须校验正文和截图指纹，不能用同一个幂等键提交不同内容。

### 4.2 完成与完成说明

- 开发者把正常需求切换为 `COMPLETED` 时，界面打开完成确认窗口。
- 确认窗口允许填写完成说明和粘贴截图；两者都为空时仍可完成。
- 完成状态、说明和截图作为一次原子命令提交，使用需求版本号防止旧页面覆盖新状态。
- 已有完成说明可由任意启用开发者在需求未暂停、未归档且项目存在时修改。
- 需求改回 `UNSCHEDULED` 或 `SCHEDULED` 时不删除说明或截图；客户和开发者仍能查看，开发者可继续修改。
- 暂停或归档需求只读，不能修改完成说明；恢复正常后重新允许开发者修改。
- 完成说明正文不得写入通用操作历史载荷。历史只记录“需求完成”“完成说明已更新”和附件增减等安全事件。

## 5. 界面

- 公开备注和澄清编辑区复用现有 `ScreenshotInput`，支持文件选择、拖放、本地预览、移除和在编辑区 `Ctrl+V` 粘贴。
- 公开备注和澄清消息下方使用现有 `AttachmentGallery` 展示截图和灯箱浏览。
- 开发者选择“完成”后打开对话框，包含可选说明输入、截图区、取消和确认完成按钮。
- 需求详情增加“完成说明”区。没有内容时客户看到平静空状态；开发者看到补写入口。
- 开发者编辑已有说明时可以保留、移除旧截图并添加新截图。
- 列表与详情中的进度标签统一使用：未排期灰色、已排期蓝色、完成绿色。提醒行背景色不改变进度标签本身的语义色。

## 6. HTTP 与权限边界

带截图的写操作使用有界 multipart Route Handler，不通过仅适合文本的 Server Action 上传文件。

- 路由检查登录态、同源、请求体上限和 multipart 部件上限。
- 领域服务重新读取实时账号、项目成员关系、项目状态、需求状态和版本。
- 附件下载对无权用户统一返回不存在，并设置 `private, no-store` 与 `nosniff`。
- 文件读取继续拒绝符号链接和异常文件大小。

## 7. 备份、恢复与部署

备份 manifest、恢复校验和附件一致性检查必须覆盖三类新增附件。恢复仍要求迁移 journal 与当前代码完全一致。

公网发布固定使用：

```bash
./scripts/deploy-ubuntu.sh update root@47.121.188.131 \
  --origin http://47.121.188.131:13001
```

发布硬门禁：

1. 本机只允许发布干净、已提交且已推送的 Git 修订。
2. 远端必须在迁移或停服前用当前运行版本成功生成包含 SQLite、所有受引用截图和 manifest 的完整备份。
3. 远端备份失败立即终止，不替换现有服务、不迁移、不停服。
4. `update` 只上传代码 bundle，不上传本机 SQLite、截图或备份，不调用远端数据替换流程。
5. 新代码只对远端现有数据库执行幂等迁移。
6. 迁移、启动、附件检查或健康检查失败时，使用发布前远端备份和旧镜像回滚。
7. 发布后核对容器数据 bind mount、迁移版本、登录页、附件完整性以及核心业务表数量；保护备份必须保留，不能因发布成功自动删除。

## 8. 测试与验收

- 数据库迁移测试覆盖新表、外键、唯一约束和旧数据无损升级。
- 领域测试覆盖公开备注和双方澄清截图、幂等指纹、版本冲突、实时越权、文件失败回滚和孤儿清理。
- 完成说明测试覆盖无说明完成、带说明和截图完成、之后补写、跨进度保留、暂停/归档只读和客户禁止编辑。
- 组件测试覆盖 `Ctrl+V`、预览、移除、完成确认窗口、完成说明展示和三种进度颜色。
- API 测试覆盖请求体限制、MIME 欺骗、同源检查、附件鉴权和稳定错误响应。
- 备份恢复与附件检查测试覆盖全部新增附件表。
- Playwright 覆盖开发者贴图回复、客户贴图回复、带图完成、之后补写以及列表和详情标签颜色。
- 发布前运行 Lint、类型检查、Vitest、Playwright、隔离生产构建、依赖审计、Bash 语法和部署测试。
- 公网发布后以真实浏览器验证需求详情、截图读取和完成说明，同时验证远端备份存在且业务数据未减少。
