# 需求标题设计

## 目标

为客户需求增加独立标题。列表显示标题而不是正文摘要；新建需求必须填写标题；历史无标题需求允许原提交客户补充一次标题。

## 数据兼容

- `requests` 新增可空 `title` 文本列。
- 迁移只执行 `ALTER TABLE requests ADD COLUMN title TEXT`，不复制、不重建、不覆盖现有数据。
- 历史需求的标题保持 `NULL`，以此作为一次性补录资格；标题写入后不可再获得特殊补录资格。
- 新建和普通编辑由应用层要求标题去除首尾空白后为 1 到 100 个字符。

## 权限与行为

- 新建需求必须填写标题。
- 客户仍可正常编辑本人提交且处于“正常 + 未排期”的需求；普通编辑可修改标题、正文、类型、优先级和截图。
- 对本人提交但标题为空的历史需求，客户可在任意进度和记录状态下一次性补充标题，包括已排期、完成、暂停和归档。
- 特殊补录只接受标题与版本号，不允许同时修改正文、类型、优先级或截图。
- 项目停用后仍保持客户只读，不开放标题补录。
- 开发者不能替客户填写或修改标题。
- 所有更新继续使用版本号条件更新，成功后版本加一并写入公开操作历史。

## 界面

- 新建和普通编辑表单增加必填“标题”字段。
- 历史补录页面只显示标题输入框和保存按钮，说明这是一次性补充。
- 需求列表标题区域显示 `title`；空标题显示“待补充标题”，不再从正文生成摘要。
- 本人历史无标题需求显示可用的“补充标题”入口；已有标题但当前不可编辑的需求继续显示禁用“编辑”。
- 详情页在需求编号下显示标题；无标题时显示“待补充标题”。
- 搜索继续匹配编号和正文，并新增标题匹配。

## 验证

- 数据库迁移测试证明旧数据库升级后数据不变且标题为空。
- 领域测试覆盖新建必填、普通编辑标题、一次性补录、所有允许状态、非本人/开发者/停用项目/并发版本拒绝。
- 组件和路由测试覆盖标题字段、列表展示、补录入口和标题专用载荷。
- 浏览器 E2E 覆盖新建必填标题、列表显示标题，以及历史记录补录后入口失效。
