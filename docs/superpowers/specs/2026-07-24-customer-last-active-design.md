# 客户最近活跃时间设计

**状态：** 已批准并实施
**日期：** 2026-07-24

## 1. 目标

管理员需要了解客户最近一次实际使用系统的时间，而不仅是最近一次输入用户名和密码的时间。系统保留原有 `last_login_at` 登录审计数据，新增 `last_active_at` 作为管理员账号列表的主要展示字段，不建立活动历史。

## 2. 记录规则

- 只有启用的 `CUSTOMER` 账号通过有效会话访问系统时记录活跃；开发者、无效会话和过期会话不记录。
- 客户成功登录也视为一次活跃。
- 同一客户距离上次活跃记录不足10分钟时不写入；达到或超过10分钟后，下一次有效访问更新为服务端当前时间。
- 限频条件同时由内存判断和数据库条件更新保护，多个会话或标签页并发时不会绕过10分钟窗口。
- 系统不因静态资源、健康检查或未认证请求更新该字段，也不将该字段解释为实时在线状态。

## 3. 数据迁移

`users` 表增加可为空的 `last_active_at` UTC毫秒时间戳。升级时，已有客户的 `last_active_at` 使用其已知的 `last_login_at` 初始化；没有成功登录记录的客户保持为空，开发者保持为空。迁移不改变用户、项目、需求、附件和会话数量。

## 4. 管理员界面与权限

- 账号管理表格将“最后登录”列替换为“最近活跃”。
- 客户有记录时按中国本地时间显示到分钟；没有记录时显示“从未活跃”。
- 开发者行显示“—”。
- `lastActiveAt` 只通过已有的开发者账号管理查询暴露，客户页面和公共接口不返回。

## 5. 验证与发布

测试覆盖客户首次访问、10分钟内不更新、达到10分钟后更新、开发者不记录、管理员查询和界面显示，以及迁移和备份版本。发布前必须创建并验证包含公网 SQLite、所有受引用截图和 manifest 的完整备份；备份失败立即停止。发布只更新代码并对远端数据库执行新增迁移，发布后核对数据库完整性、外键、核心数据数量、附件完整性、迁移版本、运行版本和登录页健康状态。
