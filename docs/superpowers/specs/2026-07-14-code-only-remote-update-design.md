# 远程仅代码更新设计

**状态：** 已批准  
**日期：** 2026-07-14  
**范围：** 从本机向 Ubuntu 服务器发布代码，不上传或覆盖本机 SQLite 与截图

## 1. 背景与目标

目标服务器可能无法稳定直连 GitHub。现有 `deploy` 依赖服务器 fetch，`sync` 虽能通过 Git bundle 上传代码，却同时用本机数据覆盖服务器。新增公开命令：

```bash
./scripts/deploy-ubuntu.sh update root@SERVER_IP \
  --origin http://SERVER_IP:13001
```

该命令从本机上传当前已推送的 Git 修订，保留服务器现有 SQLite、截图和数据目录，不创建或上传本机数据备份。

## 2. 行为

- `update` 接受与 `sync` 相同的 `SSH_TARGET`、`--ssh-port`、`--origin` 和 `--port`。
- 本机工作树的已跟踪文件必须干净，当前 `HEAD` 必须存在于 `origin`。
- 本机生成完整 Git bundle，通过复用的 OpenSSH ControlMaster 连接上传；密码登录只认证一次。
- 服务器从上传 bundle 检出修订，不需要访问 GitHub。
- 远端复用现有部署流程：构建新镜像、备份服务器当前数据、停服、执行必要迁移、启动和健康检查；失败时用旧镜像和远端保护备份回滚。
- `update` 不调用本机 `ops:backup`，不上传备份目录，不调用远端 `__receive-backup`，因此不会用本机数据库或截图覆盖服务器。
- “不更新数据库”指不传输或覆盖业务数据。为保证新代码与既有数据库兼容，远端仍执行幂等迁移；没有新迁移时数据库版本保持不变。
- 上传 bundle 和临时 SSH 套接字在成功或失败后清理。

## 3. 安全边界

- 保留 SSH 主机密钥校验，不保存密码、私钥或 Token。
- SSH 目标、端口、Origin、修订和远端暂存路径继续使用现有白名单验证。
- 只发布已提交且已推送的修订，避免服务器运行无法追溯的代码。
- 不新增绕过远端备份、迁移锁、健康检查或回滚的捷径。

## 4. 文档与测试

- `--help`、README 和运维手册同时展示 `update`，明确它与 `sync` 的数据边界。
- 部署测试证明 `update` 走 Git bundle、复用 SSH 连接，并且其路径不执行本机备份或远端数据恢复。
- 运行 Vitest、Bash 语法检查、ShellCheck、类型检查、Lint、生产构建和依赖审计。
- 在真实服务器执行 `update`，对比发布前后业务表数量和截图一致性，并确认修订号、容器挂载和登录页健康状态。
