# 需求列表表头排序设计

日期：2026-07-14

## 1. 目标

需求列表支持通过点击表头改变完整结果集的排序，并让当前排序在筛选、翻页和刷新后保持一致。排序必须在服务端查询和分页之前完成，不能只重排当前页面。

## 2. 范围

- 覆盖 `/requests` 中的需求表格。
- 支持编号与需求、项目、提交人、类型、优先级、进度、记录状态和更新时间表头。
- 不改变开发者提问的提醒状态、分区位置和置顶规则。
- 不增加数据库字段或迁移。

## 3. 默认排序

未指定表头排序时，需求按以下稳定规则排列：

1. 保留现有角色提醒优先级：客户待回复记录优先于其他正常记录，正常记录优先于已暂停记录；归档仍默认隐藏。
2. 同一提醒和记录状态组内按进度排列：`已排期 → 未排期 → 完成`。
3. 同一进度内按优先级排列：`加急 → 重要 → 普通`。
4. 同一优先级内按更新时间倒序，最新记录优先。
5. 更新时间相同时按需求 ID 倒序，保证分页稳定。

开发者同样使用进度、优先级、更新时间和 ID 的默认组合；客户额外保留待回复与记录状态的前置分组。

## 4. 表头交互

- 可排序表头使用按钮语义，不把整个单元格伪装成普通链接。
- 当前字段显示升序或降序图标，并通过 `aria-sort` 暴露排序状态。
- 首次点击字段使用该字段的自然默认方向，再次点击切换方向。
- 自然默认方向：编号、项目、提交人、类型、进度和记录状态为升序；优先级为高到低；更新时间为新到旧。
- 用户选择表头后使用单字段主排序，并以需求 ID 同方向作为稳定兜底；客户待回复和记录状态分组仍保持最高优先级。
- 排序字段和方向写入 `sort`、`direction` 查询参数。切换排序时页码回到第一页，同时保留搜索和筛选条件。
- 清除筛选返回无排序参数的默认组合排序。

## 5. 领域顺序

枚举字段不按数据库英文值或中文显示文字的字典序排序：

| 字段 | 升序业务顺序 |
| --- | --- |
| 优先级 | 加急、重要、普通 |
| 进度 | 已排期、未排期、完成 |
| 记录状态 | 正常、已暂停、已归档 |
| 类型 | Bug、功能变更、新增功能 |

降序为对应业务顺序的反向。

## 6. 数据流与校验

1. 页面只接受白名单内的排序字段和 `asc`、`desc` 方向；无效值回退到默认组合排序。
2. 页面将合法排序传给 `listRequests`。
3. 查询层使用固定 SQL 表达式映射枚举顺序，在 `LIMIT` 和 `OFFSET` 之前完成排序。
4. 表头链接合并现有查询参数，移除旧页码并写入新的排序参数。
5. 分页链接继续携带当前排序参数。

## 7. 响应式与视觉

- 桌面端表头按钮沿用当前表格排版，文字与方向图标保持紧凑，不改变列宽。
- 只在当前排序字段展示明确方向，其他表头在悬停和键盘聚焦时显示可操作状态。
- 移动端继续隐藏表头；行内容和顺序由服务端结果决定，不增加占用空间的移动端排序栏。
- 图标使用现有 Lucide 图标，并为非文字信息提供无障碍名称。

## 8. 测试与验收

- 查询测试覆盖默认进度、优先级、更新时间和稳定 ID 顺序。
- 查询测试覆盖所有可排序字段、两个方向、客户提醒置顶、筛选后排序和分页稳定性。
- 组件测试覆盖表头链接、方向切换、`aria-sort`、参数保留和页码重置。
- 页面/E2E 测试覆盖点击表头后 URL、行顺序、翻页或刷新后的排序保持。
- 运行单元与集成测试、类型检查、Lint、生产构建和 Playwright 浏览器验收。

## 9. 文档对齐

实现时同步更新产品规则、用户手册、测试说明和需求追踪，删除“开发者默认仅按更新时间倒序”等过时描述。
