# 开发者提问功能设计

## 目标

为开发者增加一种独立于客户需求的项目沟通记录。开发者可针对项目主动提出问题、想法或设计思路，统一显示为“开发者提问”。客户必须在统一列表中明显看到待回复提问；客户回复后，提醒转移给开发者，开发者确认已查看后恢复普通显示。

本功能必须保持 RequestManager 的简单原则：不引入指派人、通知中心、审批、排期或额外工作台。

## 已确认规则

- 记录统一称为“开发者提问”，不再细分问题、想法或设计类型。
- 开发者创建时选择一个启用项目，填写正文，可粘贴或选择截图。
- 项目下所有启用且已完成首次改密的客户均可查看和回复。
- 任意一位项目客户回复后，所有项目客户的待回复提醒同时清除。
- 所有开发者均可查看、继续追问和确认已查看。
- 初始提问、开发者追问和客户回复都支持正文与截图。
- 每次消息正文去除首尾空白后为 1 到 10,000 个字符；正文必填，不能只发截图。
- 每次最多 8 张截图，单张最大 10 MiB，总计最大 30 MiB；格式为 PNG、JPEG 或 WebP。
- 项目停用后，历史提问仍可查看，但不能创建、回复、追问或标记已查看。
- 记录不使用需求类型、优先级、排期、暂停或归档。

## 状态机

`developer_questions.attention_status` 只允许以下三个值：

- `WAITING_CUSTOMER`：开发者刚创建或再次追问。客户侧着色并置顶。
- `WAITING_DEVELOPER`：任一项目客户回复。开发者侧着色并置顶。
- `SEEN`：开发者手动确认已查看。双方均普通显示。

转换规则：

1. 创建提问固定进入 `WAITING_CUSTOMER`。
2. 开发者发送任意后续追问后进入 `WAITING_CUSTOMER`，即使此前是 `WAITING_DEVELOPER` 或 `SEEN`。
3. 客户发送任意回复后进入 `WAITING_DEVELOPER`，即使此前已是 `SEEN`。
4. 只有开发者可将 `WAITING_DEVELOPER` 变为 `SEEN`。
5. 对 `WAITING_CUSTOMER` 或 `SEEN` 重复执行“标记已查看”返回状态冲突，不做静默成功。
6. 所有转换与消息写入使用同一 SQLite immediate transaction，并通过 `version` 防止旧页面覆盖新状态。

状态表达的是角色当前需要处理的事项，而不是永久完成状态。提问保留完整消息历史，任何新消息都可再次激活相应角色的提醒。

## 数据模型

### `developer_questions`

- `id`：自增主键，展示编号为 `ASK-000001`。
- `project_id`：所属项目外键。
- `created_by_id`：发起开发者外键。
- `content`：初始正文。
- `attention_status`：提醒状态。
- `version`：并发版本，初始为 1。
- `idempotency_key`：创建幂等键；与创建者组成唯一约束。
- `create_payload_fingerprint`：项目、正文和截图元数据指纹，防止幂等键复用到不同内容。
- `created_at`、`updated_at`。

索引覆盖项目、更新时间与状态排序；数据库检查约束限定状态和正版本号。

### `developer_question_messages`

- `id`：自增主键。
- `question_id`：提问外键。
- `author_id`、`author_role`：消息作者及写入时角色快照。
- `content`：回复或追问正文。
- `idempotency_key`：与作者组成唯一约束。
- `created_at`。

角色快照只允许 `CUSTOMER` 或 `DEVELOPER`。消息查询按 `created_at, id` 正序。

### `developer_question_attachments`

- `id`：自增主键。
- `question_id`：提问外键，便于统一授权和清理。
- `message_id`：可空；为空表示初始提问截图，非空表示消息截图。
- `uploaded_by_id`、`storage_name`、`original_name`、`mime_type`、`size_bytes`、`sha256`、`created_at`。

存储名全局唯一。应用必须验证非空 `message_id` 确实属于同一 `question_id`。截图继续存放在既有上传根目录，但通过独立 API 路由鉴权，不能复用需求附件 ID 语义。

## 权限

### 开发者

- 查看所有提问，包括停用项目中的历史提问。
- 仅能在启用项目中新建提问。
- 仅能在启用项目中追问。
- 仅能在启用项目且状态为 `WAITING_DEVELOPER` 时标记已查看。

### 客户

- 仅能查看自己当前所属项目的提问。
- 仅能在启用项目中回复。
- 同一项目任意客户回复都触发全项目客户提醒清除。
- 猜测其他项目的 `ASK-*`、消息或截图 URL 必须返回不存在或无权访问，不泄露记录是否存在。

用户被停用或处于强制改密状态时，所有查询和写入继续沿用现有拒绝规则。

## 页面与路由

- `/questions/new`：仅开发者可访问。包含项目选择、正文和截图输入。
- `/questions/ASK-000001`：双方按权限查看详情、消息和截图。
- `/api/developer-questions`：带附件的新建接口。
- `/api/developer-questions/[questionId]/messages`：带附件的回复或追问接口。
- `/api/developer-question-attachments/[attachmentId]`：鉴权读取截图。
- Server Action 仅用于“标记已查看”等不含文件的轻量动作；所有写入继续执行来源校验。

提问详情页顶部展示编号、项目、发起人、当前状态和初始正文。消息区按时间展示作者、角色、正文和截图。页面底部只有一个回复输入区；开发者文案为“继续追问”，客户文案为“回复开发者”。开发者在 `WAITING_DEVELOPER` 时额外看到“标记已查看”。

## 统一列表

现有 `/requests` 保留为统一工作列表，数据层分别查询需求和开发者提问，再映射为带判别字段的 `WorkItemDto`。不要在 SQL 中用不透明字符串拼接模拟联合模型。

增加记录类型筛选：

- `全部`
- `客户需求`
- `开发者提问`

搜索范围：

- 需求：沿用编号和正文。
- 开发者提问：`ASK-*`、初始正文和消息正文。

项目筛选同时作用于两类记录。需求专属的类型、优先级、进度和记录状态筛选启用时，只显示需求；避免让用户误以为提问具有这些字段。

### 客户排序

1. `WAITING_CUSTOMER` 的开发者提问，按更新时间和 ID 倒序。
2. 现有待客户澄清的正常需求。
3. 其他正常记录，按更新时间倒序混排。
4. 已暂停需求。

`WAITING_CUSTOMER` 行使用暖黄色浅底、左侧强调边和“待您回复”标识。客户回复后立即恢复普通样式和正常排序。

### 开发者排序

1. `WAITING_DEVELOPER` 的开发者提问，按更新时间和 ID 倒序。
2. 其他记录按更新时间倒序混排。

`WAITING_DEVELOPER` 行使用蓝绿色浅底、左侧强调边和“客户已回复”标识。标记已查看后立即取消置顶。

提问行显示 `ASK-*`、“开发者提问”、项目、发起人、沟通状态和更新时间；需求专属列显示短横线，避免表格列宽跳动。移动端保持现有标签化行布局。

分页在合并排序后执行。第一版可在 SQLite 中分别取符合条件的候选集合后在服务层稳定合并，但必须保证总数、页数和跨页排序正确；不能分别分页后再拼接。

## 截图与文件一致性

复用现有文件签名、大小、数量、暂存、提交和清理基础设施。初始提问或消息写入失败时，数据库与文件必须一起回滚；提交成功后暂存文件必须被清理。备份、恢复和附件一致性检查必须同时覆盖需求附件与提问附件。

图片查看继续使用当前同页灯箱：黑色透明蒙版、关闭按钮、点击背景或 Escape 关闭，多图支持按钮和键盘左右键切换。

## 错误处理

- 无效编号、无权限或跨项目访问：统一返回不存在或无权访问的现有业务错误，不回传内部 ID。
- 停用项目写入：返回“停用项目中的提问为只读”。
- 版本冲突：返回“提问已更新，请刷新后重试”。
- 状态冲突：返回明确的当前状态提示，不重复写事件。
- 重复幂等请求且载荷一致：返回已有结果；载荷不同：返回冲突。
- 截图无效：沿用现有中文字段错误和 HTTP 状态。
- 文件清理失败：记录结构化服务端日志，不向客户泄露存储名或路径。

## 操作历史

新增 `developer_question_events`，记录创建、开发者追问、客户回复和标记已查看。事件均为双方可见，不记录截图存储名。详情页用业务文案展示：

- “创建了开发者提问”
- “继续追问”
- “回复了开发者”
- “标记为已查看”

消息正文不复制到事件载荷，避免重复存储和意外泄漏。

## 测试与验收

### 单元与集成

- 三态状态机和非法转换。
- 开发者创建、追问、标记已查看权限。
- 项目客户查看和回复；非项目客户 IDOR 拒绝。
- 多客户中任一回复清除客户提醒。
- 客户补充回复重新置顶开发者；开发者追问重新置顶客户。
- 停用项目只读。
- 幂等、版本冲突、消息排序和截图事务回滚。
- 客户与开发者列表的稳定排序、搜索、筛选、总数和跨页分页。
- 备份、恢复及附件检查包含新附件表。

### 浏览器 E2E

使用隔离数据库完成完整双角色流程：

1. 开发者选择项目，粘贴截图并创建提问。
2. 客户列表首行出现暖黄色 `ASK-*`，显示“待您回复”。
3. 客户进入详情，打开初始截图灯箱，粘贴截图并回复。
4. 客户返回列表，记录取消置顶和强调色。
5. 开发者列表首行出现蓝绿色 `ASK-*`，显示“客户已回复”。
6. 开发者查看客户截图并标记已查看，返回列表后取消置顶。
7. 客户继续补充回复，开发者侧重新置顶。
8. 开发者继续追问，客户侧重新置顶。
9. 同项目另一客户可见并可回复；未分配客户无法访问详情、写接口和截图。
10. 在 1440x900、390x844 和 360x800 验证列表、详情、截图灯箱和输入区无横向溢出或遮挡。

发布门禁为：`npm test`、`npm run typecheck`、`npm run lint`、`npm run test:e2e`、使用隔离数据库的 `npm run build`、`npm audit`。最后必须重启本地生产服务，并用浏览器对 `http://192.168.2.45:13001` 实测开发者和客户完整流程。

## 非目标

- 不增加指派人、通知中心、邮件、站内信或未读计数。
- 不支持提问删除、暂停、归档、优先级或排期。
- 不提供富文本编辑器、附件下载管理或非图片文件。
- 不把开发者提问转换为客户需求；需要正式需求时仍由客户另行提交。
