# 客户重新打开已完成需求设计

**状态：** 已实施  
**日期：** 2026-07-22

## 1. 目标

允许客户把自己提交且已经完成的需求重新打开，使需求回到“未排期”，等待开发者重新评估。重新打开原因必须填写，可以粘贴截图；原完成说明和完成截图始终保留。

## 2. 使用场景

- 客户验收后发现问题仍存在。
- 已交付结果只覆盖了部分需求。
- 完成后出现新的复现信息，需要开发者继续处理。
- 客户误将验收结果理解为最终完成，需要恢复需求。

本功能不是撤销或删除完成记录。操作历史仍保留原完成过程，并新增从“完成”到“未排期”的状态变化。

## 3. 操作与界面

### 3.1 入口

客户仅在同时满足以下条件时看到“重新打开”按钮：

- 当前客户是需求提交人。
- 需求进度为 `COMPLETED`。
- 记录状态为 `ACTIVE`。
- 所属项目仍处于启用状态。

按钮使用恢复图标和“重新打开”文字，仅放在需求详情页的操作区域，列表页不显示。其他客户、暂停需求、归档需求和停用项目中的需求不显示可用入口。

### 3.2 确认窗口

点击“重新打开”后显示确认窗口，内容包括：

- 标题：“重新打开这条需求”
- 提示：“重新打开后，需求将回到‘未排期’，等待开发者重新评估。原完成说明和截图会继续保留。”
- 必填的“重新打开原因”文本框，最多 10,000 字。
- 截图区，支持选择、拖放和在文本框内使用 `Ctrl+V` 粘贴 PNG、JPEG、WebP。
- “取消”和“确认重新打开”按钮。

原因为空时确认按钮不可用，同时服务端继续执行必填校验。截图沿用现有限制：最多 8 张、单张不超过 10 MiB、合计不超过 30 MiB。

### 3.3 成功结果

操作成功后：

- 进度从 `COMPLETED` 变为 `UNSCHEDULED`。
- 页面刷新并显示灰色“未排期”标签。
- 原完成说明及其截图不变。
- 原因和截图出现在“确认与澄清”区域，消息标签显示“客户重新打开”。
- 该区域因为已有内容而继续使用橙色标题和图标。
- 客户重新获得现有“编辑”能力，可以继续修改未排期需求。
- 操作历史显示客户把进度从“完成”改为“未排期”。

## 4. 数据设计

复用现有澄清消息及澄清截图模型，不新增独立的重新打开区域：

- `clarification_messages` 增加 `message_kind` 字段。
- 现有数据默认值为 `CONVERSATION`。
- 重新打开原因使用 `REOPEN_REASON`。
- 截图继续写入 `clarification_message_attachments`。

查询 DTO 增加消息类型。普通开发者提问和客户回复保持现有标签；`REOPEN_REASON` 在双方视图中固定显示为“客户重新打开”。这种设计让原因处于已有沟通上下文中，同时继续复用附件鉴权、灯箱、备份和完整性检查。

## 5. 服务与事务

新增独立的重新打开领域命令和有界 multipart 路由：

```text
POST /api/requests/{requestId}/reopen
```

请求包含：

- `expectedVersion`
- `reason`
- `idempotencyKey`
- `attachments[]`

服务端在 SQLite `IMMEDIATE` 事务中完成：

1. 重新读取实时账号并确认账号启用、角色为客户且无需强制改密。
2. 确认客户是需求提交人，并具有启用项目的成员资格。
3. 确认记录正常、进度确实为完成、版本匹配。
4. 写入 `REOPEN_REASON` 澄清消息及截图元数据。
5. 把进度改为未排期，把 `needsCustomerReply` 设为 `false`，版本加一并更新时间。
6. 写入公开的 `PROGRESS_CHANGED` 历史事件。

文件先暂存并校验。数据库、文件提交或状态更新任一步骤失败时，数据库回滚并清理本次文件，需求保持完成状态。成功后再刷新列表和详情。

## 6. 幂等与冲突

- 客户端为每次弹窗生成随机幂等键。
- 服务端指纹包含需求、原因和截图元数据。
- 相同幂等键与相同内容重放时返回第一次成功结果，不重复写消息、截图或历史。
- 相同幂等键提交不同内容时返回冲突。
- 旧版本页面、已经被重新打开的需求或并发状态变化返回明确冲突，提示刷新后重试。

## 7. 权限与异常

- 开发者仍使用现有进度控件，不使用客户重新打开弹窗。
- 客户不能重新打开他人需求。
- 暂停、归档、项目停用或非完成需求不能重新打开。
- 越权读取截图统一返回不存在。
- 原因正文作为客户可见沟通内容展示，但不复制到通用历史事件载荷。

## 8. 测试与验收

- 迁移测试：旧澄清消息升级后类型为 `CONVERSATION`，新类型可正常保存和查询。
- 领域测试：客户成功重新打开、原因必填、截图保存、幂等重放、版本冲突、所有越权与状态拒绝。
- 原子性测试：文件或数据库失败时状态、消息和截图全部不落地。
- 保留测试：完成说明和完成截图在重新打开后保持不变。
- 组件测试：按钮可见性、弹窗文案、必填状态、粘贴截图和“客户重新打开”标签。
- Playwright：客户完成重新打开操作，开发者看到原因与截图，需求显示未排期且客户重新获得编辑入口。
- 发布前运行类型检查、Lint、Vitest、隔离生产构建和 Playwright。
