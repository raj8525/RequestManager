# 测试说明

## 自动化分层

| 层级 | 命令 | 覆盖重点 |
| --- | --- | --- |
| 单元/组件/SQLite 集成 | `npm test` | 状态机、排序、密码、会话、权限、文件验证、运维原子性、停止容器遗留 WAL 恢复、可访问组件 |
| 类型检查 | `npm run typecheck` | TypeScript 合约和服务端/客户端边界 |
| 静态检查 | `npm run lint` | React、Next.js 和代码规范 |
| 浏览器验收 | `npm run test:e2e` | 双角色主流程、IDOR、私人笔记载荷隔离、截图、澄清、响应式 |
| 生产构建 | `npm run build` | Next.js 生产编译和路由生成 |
| 依赖审计 | `npm audit` | 直接和传递依赖漏洞 |
| 部署脚本 | `bash -n scripts/deploy-ubuntu.sh`、Vitest deployment tests、ShellCheck | 参数、密钥边界、构建/备份/停服顺序、回滚、完整同步 |

Vitest 只收集 `tests/**/*.test.ts(x)`；Playwright 只收集 `e2e/*.spec.ts`。E2E 使用 `data/e2e` 隔离数据库和截图，启动器在 Next.js 打开数据库前完成重建和迁移，并拒绝与正式路径重叠。

## 浏览器验收场景

- 匿名重定向、统一登录错误、角色导航、强制改密和退出。
- 客户粘贴截图创建需求并编辑；开发者排期、公开备注、私人笔记和提问。
- 同项目另一客户看到红色置顶并回复；提醒清除；再次提问重新出现。
- 开发者创建带截图的提问；客户置顶回复后提醒转移到开发者；开发者标记已查看后取消置顶。
- 暂停、恢复、归档和归档筛选。
- 未分配客户猜测详情、写接口、管理页和截图 URL 均失败。
- 开发者 A 的私人笔记不出现在开发者 B 或客户的 HTML/RSC/网络响应。
- 1440x900、1024x768、390x844、360x800 无页面级横向溢出，管理弹窗可键盘关闭并保持在视口内。

## 剪贴板证据边界

自动化生命周期测试使用浏览器 `ClipboardEvent` 和包含真实 PNG 字节的 `DataTransfer`，验证应用的粘贴事件、预览、服务端文件签名和最终鉴权读取。这证明应用逻辑，但不等同于具体操作系统的剪贴板集成。

发布前人工烟测应在目标操作系统中：复制一张真实截图，将焦点放在需求正文，使用系统粘贴快捷键，确认预览出现并能提交。移动端还要验证系统文件选择器。

## 完整发布门禁

```bash
npm run lint
npm run typecheck
npm test
npm run test:e2e
npm run build
npm audit
```

门禁之外还需要隔离路径的备份/恢复演练和当前生产构建的桌面、移动浏览器烟测。测试失败时不要复用可能指向正式数据的服务器进程。

## Docker 部署验证

`tests/deployment` 的容器契约和伪命令测试不操作正式服务器，主要证明输入拒绝、安全约束与生命周期顺序，包括在 Origin 不可访问时从上传的 Git bundle 更新检出、自动沿用旧容器的 bind mount 数据目录，以及启动前精确比对恢复后的 SQLite 与上传快照。发布部署脚本还必须在可用 Docker daemon 上真实构建镜像，并使用隔离的数据目录和非正式端口完成：迁移空库、初始化开发者、启动登录页、创建完整备份、停服恢复、截图一致性检查，以及对同一容器执行停止、启动后连续访问登录页。重启烟测还要确认新进程替换旧的 PID/进程启动标识锁，而不是误报已有应用占用数据库。

若当前环境没有 Docker 或 Ubuntu 虚拟机，必须明确记录未执行的真实门禁；不能用文本契约测试替代“已在所有 Ubuntu 上验证”的声明。服务器正式同步前仍应先用非正式数据完成一次 SSH 上传演练。
